# 瓶颈判断报告

## 结论先行

当前卡顿的主瓶颈不在服务器转码，而在浏览器端：

- H265 软解成本高
- 分辨率偏大
- 渲染链路容易抖动
- 部分源类型和渲染方式不兼容

对你这条 `RTMP -> ZLM raw_flv -> 浏览器` 链路，优先问题是浏览器侧。

## 已观测事实

### 1. 源头信息

后端探测到的源：

- `source_video_codec = hevc`
- `source_width = 2304`
- `source_height = 1296`
- `raw_flv_url` 已可用
- `recommended_profile = raw_h265`

这意味着服务端没有卡在拉流或转码生成上，原始 H265 直出是可用的。

### 2. 浏览器播放表现

对这条 RTMP H265 源，前端实测出现：

- 绿屏
- 画面明显卡断
- `fps` 很低
- `p95` 帧间隔很高

在当前一次有效实测里，状态大致是：

- `fps ≈ 9`
- `frameIntervalP95Ms ≈ 186`

这说明不是“平均慢一点”，而是“节奏很不稳”。

### 3. 其他路径对比

对确认可用的 HTTP-FLV H265 源，前端实测曾达到：

- `fps ≈ 24`
- `p95 ≈ 41.8ms`

说明浏览器端不是完全不行，而是：

- 源格式
- 分辨率
- 播放链路
- 渲染方式

共同决定体感。

### 4. 错误路径确认

我们自写的 WebGL YUV 渲染在这条 RTMP 源上会绿屏，说明：

- 解码结果虽然出来了
- 但渲染层对这路源不够稳

这不是服务器瓶颈，而是浏览器渲染链路不兼容。

## 瓶颈排序

按当前证据，优先级大致是：

1. 浏览器 H265 软解 CPU
2. 大分辨率 `2304x1296`
3. 渲染链路不稳定/不兼容
4. 主线程调度和 GC 抖动
5. 上游网络/推流抖动

## 为什么不是服务器先背锅

因为后端已经把这条流识别出来并提供了 raw H265 直出：

- 不需要默认服务器转码
- raw_flv 可用
- 服务端状态是 `playing`

所以服务端不是当前第一瓶颈。

## 多线程 Worker 是否能解决

只能部分缓解。

- 多路流：可以“一路一个 Worker”
- 单路 H265：不能靠普通 Worker 随便拆帧

真正的单路多线程，需要：

- WASM pthreads
- `SharedArrayBuffer`
- `COOP/COEP`
- 解码器内部支持并行

这属于更重的架构升级，不是简单改几个参数能解决。

## 当前建议

### 短期

- 保留浏览器优先策略
- 不默认触发服务器 H264 转码
- 对 RTMP/RTSP 和 HTTP-FLV 分别走不同播放链路
- 继续采集 `decodedFps / renderedFps / p95 / droppedFrames`

## 已落地的诊断指标

前端现在会在 H265 浏览器直出时显示“诊断”面板，用来区分到底卡在哪里。

### 指标含义

- `解码`: 浏览器每秒实际解出来的视频帧数。
- `渲染`: 每秒真正画到页面上的帧数。
- `耗时`: 单帧绘制平均耗时。
- `P95`: 最近一段时间帧间隔的 95 分位，越高越容易肉眼卡顿。
- `丢帧`: 前端为了追实时主动丢掉的帧。
- `排队`: 前端渲染队列里积压的帧。
- `长任务`: 最近 10 秒浏览器主线程超过 50ms 的阻塞次数。
- `内存`: 当前 JS 堆内存估算值，Chrome 下可用。
- `分辨率`: 后端探测到的源分辨率。
- `链路`: 当前使用的浏览器播放路径。

### 判断规则

- `解码低、渲染也低`: 大概率是 H265 WASM 软解 CPU 吃不住。
- `解码高、渲染低`: 解码出来了，但绘制/页面主线程追不上。
- `耗时 > 18ms`: 单帧绘制偏重，60fps 肯定吃紧，25fps 也容易抖。
- `P95 > 120ms`: 画面节奏明显不稳，体感就是 PPT。
- `长任务很多`: 页面主线程或 GC 阻塞明显，需要把更多工作移出主线程。
- `排队持续升高`: 播放链路有积压，应该丢旧帧保实时。

### 当前策略调整

已经取消“低 FPS 自动切服务器 H264 转码”。原因是这会把问题藏起来，也会违背浏览器优先的核心目标。

现在只有硬失败才触发服务器兜底，例如：

- 首帧超时
- 播放器初始化失败
- 解封装失败
- 浏览器完全不支持当前链路

### 中期

- 做性能面板，拆分解码和渲染成本
- 先确认到底是解码慢还是绘制慢
- 如果是解码慢，再评估真正的多线程 WASM 解码器

### 长期

- 引入支持 pthreads 的 H265 WASM 解码器
- 或优先依赖浏览器硬解能力
- 再不行才启用服务端低清兜底

## 一句话判断

当前卡顿主要是**浏览器端 H265 软解 + 高分辨率 + 渲染链路不稳定**，不是服务器转码压力。
