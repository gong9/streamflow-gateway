# 前端 H265 播放优化记录

## 背景

项目目标是尽量把 H265/HEVC 解码压力放到浏览器端，避免服务器为每一路 H265 源做 H265 -> H264 转码。

当前主路径：

- 服务端探测源编码。
- H265/HEVC 源优先走原始 HTTP-FLV 直出。
- 浏览器使用 Jessibuca 4 / JV4 SIMD WASM 解码。
- 只有浏览器播放失败或过慢时，才回退到服务器 H264 兼容转码。

## 这次发现的问题

页面显示“原始直出 / H265 / 直连播放”只能说明服务器没有转码，不能说明播放一定顺畅。

真实问题在前端：

- 解码回调到达节奏不一定均匀。
- 如果每解出一帧就立刻画，画面可能出现突发、停顿、跳帧。
- 之前用 `requestAnimationFrame` 做 25fps 节奏控制时，在 60Hz 屏幕上 40ms 目标容易落到约 50ms，实际只剩约 20fps。

## 修复策略

新增前端 `PacedRenderer`：

- 解码和渲染分离。
- 解码后的帧先进入短队列。
- 使用约 40ms 定时节奏绘制，贴近 25fps 摄像头源。
- 队列过长时丢弃旧帧，保留新画面。
- 监控场景优先保证实时性，不追求把每一帧都播完。

取舍：

- 会主动丢旧帧。
- 但能降低“攒几秒再跳一帧”的 PPT 观感。
- 对摄像头预览更合适，因为实时画面比历史帧完整性更重要。

## 实测结果

测试源：

```text
https://example.test/live/h265.flv?codeType=H265
```

源特征：

- H265/HEVC
- 1920x1080
- 25fps

修复前一版：

- 前端渲染约 20fps
- p95 帧间隔约 51.6ms
- 观察到明显不稳

修复后：

- 前端渲染约 24fps
- p95 帧间隔约 42.6ms
- 客户端丢旧帧约 2 帧/采样窗口
- 页面显示“流畅度：顺畅”

## 新增指标

前端 H265 播放会统计：

- `fps`：实际渲染 FPS，不再只看解码回调。
- `decodedFps`：浏览器 WASM 解码速度。
- `droppedFrames`：前端为了实时性主动丢弃的旧帧。
- `frameIntervalP95Ms`：帧间隔 p95，用来判断肉眼卡顿。
- `renderCostMs`：单帧绘制耗时。
- `pipeline`：当前播放链路，例如 `worker-simd-paced`。

## 当前边界

这套优化能解决“帧呈现节奏不稳”的问题，但不能突破所有浏览器端性能限制。

仍然可能卡顿的情况：

- 客户端机器性能较弱。
- 1080p H265 码率过高。
- 浏览器 WASM 解码本身跑不满 25fps。
- 网络流本身抖动严重。
- 同一个浏览器同时打开多路 H265 1080p。

如果需要继续突破，可以考虑：

- 服务端提供低分辨率原始流或子码流。
- 前端允许选择 540p/720p/1080p。
- 探索 OffscreenCanvas Worker 内渲染，减少主线程绘制压力。
- 对 H265 源做更细的能力检测：机器弱时自动建议轻量模式。
